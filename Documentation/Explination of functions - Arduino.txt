Moss Machine — Arduino Firmware Explanation

This document explains how the `Main.ino` Arduino firmware controls the Moss Machine card sorting hardware and communicates with the host Python application (`gui_interface_enhanced.py` and `optimized_scanner.py`).

WHAT IT DOES

The Arduino Mega firmware orchestrates automated card sorting by:
- Controlling X/Y/Z stepper motors to position the pickup head over cards and destination trays
- Operating vacuum relays to pick up and release individual cards
- Reading a VL6180X time-of-flight (ToF) distance sensor to verify successful pickup/drop
- Tracking endstops for safety and calibration (homing sequence)
- Accepting commands from the Python host via serial communication
- Reporting machine state, sensor readings, and errors back to the host
- Managing 34 physical tray locations with automatic overflow handling

Hardware summary
- Stepper motors: X, Y, Z + E0, E1 (driven via RAMPS/A4988 or similar)
- Vacuum relays: `Vacuum1` (pickup) and `Vacuum2` (release)
- VL6180X ToF sensor (I2C)
- I2C 16x2 LCD display (I2C address 0x27)
- Endstops: X/Y/Z min and max (6 total)
- Misc: LED lighting, tray count tracking

Key constants and default values (from code)
- `initial_pickup_distance = 6000` (Z steps for initial pickup move)
- `initial_drop_distance = 4000` (Z steps for drop)
- `Xcal = 350`, `Ycal = 475`, `Zcal = 140` (movement multipliers)
- `speed = 700` (microsecond delay for X/Y step pulses — higher = slower)
- `zspeed = 75`, `zespeed = 120` (Z-axis microsecond delays)
- `pickup_threshold = 40`, `release_threshold = 40` (ToF millimeter thresholds)
- `HCC = 10` (homing cycle count — used to rehome periodically)
- `buffSize = 40`, `numChars = 64` (serial buffer sizes)
- `timeout = 10` seconds (watchdog/timeout for inactivity)

Tray system and capacities
- Trays are indexed from 1..34
- Tray 33 is reserved as the **RejectCard** bin
- Tray 34 is reserved as the **OverflowTray**
- Each tray has a capacity cap of 375 cards (the firmware stops if overflow capacity is reached)

SERIAL PROTOCOL (how Python talks to Arduino)

All messages use angle brackets as framing:
- Start marker: `<`
- End marker: `>`
- Format: `<CommandName>` or `<Command,param1,param2>`

Examples from Python → Arduino:
- `<StartMachine>` — enable card sorting mode
- `<StopMachine>` — disable card sorting, return to idle
- `<tray1>` — route next card to tray group 1
- `<RejectCard>` — send card to reject bin (tray 33)
- `<HomeButton>` — run full homing calibration
- `<QuerySensors>` — request current sensor/endstop readings
- `<SetParam,speed,650>` — change movement speed parameter
- `<SetMotor,Vacuum1,1>` — turn on vacuum relay (manual control)

Arduino → Python responses:
- `<Arduino is ready>` — sent on startup and after each command completion
- `<OK,MachineStarted>` — confirms machine state change
- `<Error,MachineNotStarted>` — command rejected (machine must be started first)
- `<Sensors,range=45,xmin=0,xmax=1,ymin=0,ymax=1,zmin=1,zmax=0,home=1,started=1>` — structured sensor data

Implemented commands (examples)
- `StartMachine` / `StopMachine` — switch machine state (accept or reject tray commands)
- `trayX` (where X is one of a small matching list) — assign/trip trays via `ForLoop`/`Tray` logic (host typically sends `tray1`, `tray7`, etc.)
- `CalibrateX1`, `CalibrateX2`, `CalibrateY1`, `CalibrateY2`, `HomeButton` — manual small movements or full homing
- `QuerySensors` — returns range and endstop states in a structured response
- `SetMotor,<MotorName>,<0|1>` — toggle motor/relay pins (only when machine is stopped)
- `SetParam,<param>,<value>` — change firmware parameters (only at home and when stopped)
- `QueryParams` — returns the current parameters

Core functions

getDataFromPC()
- Reads bytes from `Serial` into an input buffer and detects message framing using `<` and `>` markers.
- When an end marker is received the buffer is null-terminated and `parseData()` is invoked.

parseData()
- Tokenizes the received CSV-like payload and stores the first token into `messageFromPC` for command handling.

DetermineAction()
- Main request router. Takes `messageFromPC` and:
  - Handles `StartMachine` and `StopMachine` (changes `machineStarted` flag and replies `<OK,MachineStarted>` or `<OK,MachineStopped>`)
  - Runs `ForLoop()` to assign or route trays when tray commands are received (requires `machineStarted`)
  - Validates state for motor/param commands (many are only allowed when machine is stopped and at home)
  - Implements `SetMotor`, `SetParam`, `QueryParams`, and `QuerySensors` responses

ForLoop(start, end)
- Iterates trays between `start` and `end` (these ranges are defined in arrays in the firmware) to find an assigned tray slot or an empty slot.
- If it finds a slot assigned to the incoming token it calls `Tray(i)` to execute movement.
- If it reaches tray 34 and that tray is below capacity it uses the overflow tray; if overflow is full it calls `StopMachine()` to halt operations.

Tray(var)
- Executes the end-to-end movement to pick a card and route it to tray `var`:
  1. Move Z to `initial_pickup_distance`
  2. Read ToF sensor and compute pick steps using `(range[0] + range[1]) / 2 * Zcal`
  3. Call `pick()` to engage vacuum and lift the card
  4. Compute X/Y move from `X` and `Y` coordinate arrays and `xOffsets`/`yOffsets` tables
  5. Move to target coordinate with `Move4()` and `Move1()` helper functions
  6. Release with `pick(..., Release=1)` and verify drop via `ReadRange()`
  7. Return to home position and optionally run a course-correction step
  8. Increment tray counters and `upcount` and re-home every `HCC` picks

pick(steps, Release)
- Controls vacuum relays for pickup (Vacuum1) and release (Vacuum2).
- Implements retry logic: if pickup fails (ToF doesn't indicate expected distance) it retries up to 10 times then stops the machine.

Move4(dir1, dir3, steps1, steps2) and Move1(dir, steps, speed1)
- Low-level motor stepping routines. `Move4` steps both X and Y axes concurrently by pulsing step pins with delays controlled by `speed`.
- `Move1` performs Z-axis moves for a specified number of step pulses.

ReadRange(index)
- Reads the VL6180X two times and stores values in the `range[]` array. Used for pickup/drop verification and for `QuerySensors`.

SAFETY AND ERROR HANDLING

Sensor error detection:
- The firmware continuously monitors VL6180X status codes
- Recognized errors: ECE failure, convergence errors, SNR issues, range overflow/underflow
- Error messages are displayed on the I2C LCD and sent to the serial console
- The sensor includes automatic retry and recovery logic

Pickup retry mechanism:
- If a card isn't detected after pickup (ToF reading > pickup_threshold), the firmware retries
- Maximum 10 retry attempts before calling `StopMachine()` to prevent damage
- This prevents the machine from attempting to move a card that wasn't successfully grabbed

Overflow protection:
- Each tray has a 375-card capacity limit tracked in `CountArray[]`
- When overflow tray (34) reaches capacity, the machine automatically stops
- Prevents mechanical jams and card damage from overfilled trays

Endstop monitoring:
- Six endstops (X/Y/Z min/max) are read during homing and idle states
- Used primarily for calibration; not actively enforced during high-speed sorting
- Future enhancement: real-time endstop checking during motion to prevent crashes

TUNING AND BEST PRACTICES

Parameter tuning (modify these to match your hardware):
- `Xcal`, `Ycal`, `Zcal` — steps per unit distance (calibrate by measuring actual movement)
- `speed` — X/Y step delay in microseconds (higher = slower, increase if motors skip steps)
- `zspeed`, `zespeed` — Z-axis speeds for pickup and return (tune for card weight)
- `pickup_threshold`, `release_threshold` — ToF distance thresholds in mm (adjust for card thickness)

Homing cycle (`HCC`):
- Default: re-home every 10 cards to correct accumulated positioning errors
- Increase for faster operation; decrease for better accuracy
- The machine tracks `upcount` and calls `Homemachine()` when `upcount % HCC == 0`

Security considerations:
- Anyone with serial access can control the machine — keep the USB port physically secure
- The Python GUI should be the only process communicating with the Arduino
- No authentication is implemented in the firmware (simple command protocol)

Testing workflow:
1. Test homing sequence with no cards loaded
2. Verify vacuum pickup with a single test card
3. Run small batches (10-20 cards) and verify tray routing
4. Gradually increase speed parameters after confirming mechanical stability
5. Use inexpensive bulk commons for initial testing, not valuable cards

TYPICAL OPERATION SEQUENCE

1. Arduino powers on → runs `setup()` → initializes hardware → runs `Homemachine()` → sends `<Arduino is ready>`
2. Python GUI connects via serial port (typically COM3 at 9600 baud)
3. User clicks "START MACHINE" button → GUI sends `<StartMachine>` → Arduino sets `machineStarted = true`
4. Camera detects a card → `optimized_scanner.py` identifies it → determines bin via sorting mode
5. GUI sends bin command like `<tray7>` → Arduino calls `ForLoop(7, 13)` → finds/assigns tray → calls `Tray(N)`
6. `Tray(N)` sequence:
   - Move Z down, read ToF sensor, calculate precise pickup distance
   - Engage Vacuum1, lift card, verify pickup via ToF
   - Move X/Y to destination tray coordinates
   - Lower Z, release Vacuum2, verify drop via ToF
   - Return to home position, apply course corrections
7. Arduino sends `<Arduino is ready>` → GUI can send next card
8. After every 10 cards (`HCC`), Arduino automatically re-homes to maintain accuracy
9. If overflow tray fills (375 cards) → Arduino calls `StopMachine()` → displays "Empty and reset" on LCD

HOW IT INTEGRATES WITH PYTHON

The Arduino is a "dumb" actuator — it doesn't recognize cards or make sorting decisions. The Python components handle:
- `optimized_scanner.py`: image recognition via perceptual hashing, database lookups, bin assignment logic
- `gui_interface_enhanced.py`: camera capture, user interface, serial command orchestration, parameter management

The division of labor:
- Arduino = physical control, safety, tray tracking, sensor reading
- Python = vision, intelligence, user interaction, database queries
